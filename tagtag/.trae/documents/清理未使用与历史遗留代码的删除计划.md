## 目标
- 系统性删除未使用、冗余或已被新实现替代的代码，降低复杂度与维护成本
- 保持编译与运行稳定，逐项验证

## 删除清单（第一批）
1. PageQuery 旧白名单方法
- 删除：`filterSortByWhitelist(...)`、`assertSortWhitelist(...)`
- 位置：`tagtag-common/src/main/java/dev/tagtag/common/model/PageQuery.java`
- 理由：已统一使用 `PageQuery.buildOrderBy(...)` + 实体元数据解析，不再从服务层调用旧方法

2. Pages 未使用方法
- 删除：`toPage(...)`、`toPageNormalized(...)`、`buildOrderBy(...)`（重载，非被调用版本）
- 位置：`tagtag-framework/src/main/java/dev/tagtag/framework/util/Pages.java`
- 理由：服务层已全面切换到 `toPageAndOrder(...)` 与 `selectPage(...)`

3. OrderByBuilder 旧 API
- 删除：`mapping(...)` 与 `build(List<SortField>, ...)` 及相关私有辅助
- 位置：`tagtag-framework/src/main/java/dev/tagtag/framework/util/OrderByBuilder.java`
- 理由：排序已统一通过 `PageQuery.buildOrderBy(...)` 与 `mpResolver(...)`，旧 API无人调用

## 保留项（暂不删除）
- PageProperties 白名单：当前封装仍依赖其集合（`Pages.selectPage(...)`）；若后续改为实体注解方案，可再删除
- DevToolsController（dev profile）：便于本地开发，非生产影响
- GlobalConstants 中未使用常量：不影响运行，后续二次清理时可移除

## 验证步骤
- 编译：`mvn -DskipTests compile`
- 运行冒烟：关键接口分页与排序（用户、部门、角色、菜单）
- 关注点：
  - 排序 SQL 是否稳定生成（默认 id ASC）
  - 仍能正确处理 `String`/`ResponseEntity` 响应封装边界
  - traceId 过滤器正常注入与清理

## 风险与回滚
- 若发现某处仍隐式依赖被删方法，回滚该删除并评估调用链
- 删除仅限未引用方法，不涉及业务逻辑分支；风险极低

## 交付
- 执行删除 → 编译验证 → 简单分页接口冒烟（四模块）
- 输出清理差异与验证结果
